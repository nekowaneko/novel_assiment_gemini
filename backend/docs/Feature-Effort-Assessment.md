# 功能工時評估 (Feature Effort Assessment)
## 專案名稱： AI 寫作伴侶 (AI Writing Companion)

### 1. 評估標準
以下評估採用 1 到 5 的分數制，分數越高代表所需工時越長，開發難度也越高。

| 分數 | 說明 |
| :-- | :-- |
| 1 | 幾乎不需額外開發工時，主要為既有功能的利用或使用者操作。 |
| 2 | 簡單功能，邏輯直接，可在短時間內完成。 |
| 3 | 中等難度，涉及多個元件的互動或簡單的演算法。 |
| 4 | 複雜功能，是專案的核心，需要花費較多時間處理邏輯與邊界條件。 |
| 5 | 極高難度，可能涉及全新的技術探索或複雜的協作流程。 |

### 2. 功能評估列表

| 功能 | 評估分數 (1-5) | 評估理由 |
| :-- | :-- | :--|
| 1. 使用者上傳、編輯與儲存小說內容 (.docx)。 | 2/5 | 這是專案的基礎功能。Streamlit 和 python-docx 函式庫都提供了現成且穩定的工具，使讀取與寫入檔案的邏輯相對簡單。主要工作是將這些工具整合進 UI 流程中。 |
| 2. 讀者AI給出讀後評價。 | 2/5 | 這是一個相對單純的單次 API 呼叫。開發者只需將使用者選取的文本片段作為輸入，直接傳給 Google Gemini API，然後將回覆顯示在介面上。不需要管理複雜的狀態或對話歷史，因此難度較低。 |
| 3. 與編劇AI討論後續情節的設計。 | 4/5 | 這是本專案的核心功能與主要難點。雖然有 LangChain 的協助，但開發者仍需妥善管理多輪對話的上下文。這涉及：<br/>- 建立流暢的聊天介面。<br/>- 確保每次對話都能正確地將歷史紀錄傳給 LLM。<br/>- 處理非同步呼叫與潛在的 API 錯誤。<br/>這個功能需要投入最多的時間來確保其穩定性與流暢度。 |
| 4. 將與AI的對話內容複製，用以修改小說內容。 | 1/5 | 此功能主要仰賴瀏覽器本身提供的選取與複製功能。開發者唯一需要確保的是 AI 的回覆是可選取的文本。這是一個使用者層面的操作，程式碼中幾乎不需要任何額外邏輯。 |

### 3. 開發建議
建議您依循以下順序進行開發，以確保專案能循序漸進地完成：

1. 優先完成功能 1、2、4：這三個功能開發難度低，能快速建構一個可運行的最小可行產品 (MVP)，讓您能盡早開始測試核心使用者體驗。

2. 專注於功能 3：在 MVP 基礎上，再投入時間專注於最複雜的多輪對話功能。

這樣能避免一開始就陷入複雜的 LLM 邏輯，讓您的開發旅程更順暢。
