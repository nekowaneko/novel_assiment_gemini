# 市場調查與轉型分析：AI 寫作伴侶 (Companion App Pivot)

本文件針對將產品轉型為「AI 寫作伴侶」進行市場與技術可行性分析，重點比較「並排顯示」與「自動讀取」兩種技術路徑，並定義目標客群與 AI 角色。

## 1. 評分標準定義 (Scoring Standards)

### 1.1 開發困難指數 (Development Difficulty Index)
*範圍：0 (極易) - 5 (極難)*
*   **0-1**: 使用現有成熟技術，無須額外學習成本，風險極低。
*   **2-3**: 需要一定程度的技術整合或學習新框架，有可控的技術風險。
*   **4-5**: 涉及作業系統底層 API、第三方軟體深度整合 (Plugin/Add-in)，或逆向工程，除錯困難且維護成本高。

### 1.2 開發時長指數 (Development Duration Index)
*範圍：0 (即時) - 5 (長期)*
*   **0-1**: < 1 週。可快速製作原型或 MVP。
*   **2-3**: 1 - 4 週。需要完整的開發週期。
*   **4-5**: > 1 個月。需要大量的研發、測試與相容性調整。

### 1.3 受訪者需求排序 (Respondent Demand Priority)
*範圍：(-3) (優先滿足) - 0 (不優先滿足)*
*   *此指數代表使用者的「痛點程度」或「需求迫切度」。負值越大，代表需求越強烈。*
*   **-3 (Must Have)**: 核心痛點，沒有此功能使用者可能會拒絕使用 (Deal Breaker)。
*   **-2 (Should Have)**: 重要需求，能顯著提升體驗，但非絕對必要。
*   **-1 (Could Have)**: 加分項目，有的話很好，沒有也能接受。
*   **0 (Won't Have)**: 使用者不在意，或對體驗影響極小。

## 2. 技術路徑比較：並排顯示 vs. 自動讀取

| 比較項目 | 方案 A：並排顯示 (Side-by-Side / 獨立小視窗) | 方案 B：自動讀取 (automatic-read / Plugin / VSTO) |
| :--- | :--- | :--- |
| **技術描述** | 獨立視窗應用 (Floating Window)，需使用者手動複製貼上文字。視窗需置頂 (Always-on-top)。 | 開發 Word/Scrivener 專屬插件或使用 OS Accessibility API 自動讀取當前視窗內容。 |
| **開發困難指數** | **1** (低) <br> *理由：標準 Web/Electron 技術，僅需處理視窗置頂，技術門檻極低。* | **5** (高) <br> *理由：需掌握 .NET/C# (VSTO) 或 Office JS API，甚至需處理 Win32 API 勾子，且不同軟體皆需客製化開發。* |
| **開發時長指數** | **1** (短) <br> *理由：可沿用現有程式碼，僅需優化 UI 尺寸與交互。* | **5** (長) <br> *理由：需從零學習並搭建插件開發環境，且測試與發布流程繁瑣 (需簽署憑證等)。* |
| **受訪者需求排序** | **-1** (中/低) <br> *理由：雖比完全切換視窗方便，但「複製貼上」仍中斷心流，體驗不夠 "Magic"。* | **-3** (高/極高) <br> *理由：完美實現「所寫即所得」的輔助，與 Notion/Github Copilot 體驗一致，是使用者夢寐以求的功能。* |
| **綜合評估** | **MVP 階段首選**。能快速驗證 AI 核心價值 (Prompt/RAG)，風險低，且不限特定軟體 (通用性高)。 | **長期終極目標**。體驗極佳但成本極高，建議待產品核心價值 (AI 回答品質) 被驗證後再投入開發。 |

## 3. 目標客群重定義 (Target Audience Redefined)

*   **定義**：已有固定慣用寫作軟體（如 Microsoft Word, Scrivener, Obsidian 等）的重度創作者。
*   **特徵**：
    *   **軟體黏著度高**：不願意/無法更換原本的寫作工具（因為格式設定、排版習慣或軟體依賴）。
    *   **追求極致效率**：對寫作流程中的任何干擾（如視窗切換、格式跑掉）都非常敏感。
    *   **痛點**：
        1.  現有 AI 工具 (ChatGPT) 需頻繁切換視窗，打斷思路。
        2.  AI 生成內容需手動整理回原稿，格式易亂。
        3.  通用 AI 不懂其特定的寫作風格與設定。

## 4. AI 角色定位 (AI Personas)

根據 `backend/src/utils/gemini_client.py` 中的系統提示詞 (System Prompts) 定義：

### 4.1 AI 讀者 (The Reader)
*   **核心定位**：專業的讀者與評論家 (Professional Reader & Critic)。
*   **主要任務**：
    *   閱讀使用者提供的小說片段。
    *   提供包含「整體感受」、「優點」與「可改進處」的具體評價。
*   **互動風格**：
    *   **友善且具鼓勵性 (Friendly & Encouraging)**：避免過於尖銳的批評，以建設性為主。
    *   **具體且長度適中**：不空泛讚美，也不長篇大論。

### 4.2 AI 編劇 (The Screenwriter)
*   **核心定位**：才華洋溢的協作夥伴 (Collaborative Partner)。
*   **主要任務**：
    *   **腦力激盪 (Brainstorming)**：與作者共同發想情節、角色與世界觀。
    *   **情節發展 (Plot Development)**：針對卡關處提出具體的後續發展建議。
    *   **記憶與上下文 (Context Memory)**：記住過往對話與知識庫設定 (RAG)。
*   **互動風格**：
    *   **簡潔 (Concise)**：除非被要求詳細解釋，否則保持精簡。
    *   **繁體中文 (Traditional Chinese)**：主要回覆語言。
    *   **具創造力 (Creative)**：提供意想不到的點子。
