# 測試與部署文件 (Testing & Deployment Document)

## 專案名稱： AI 寫作伴侶 (AI Writing Companion)

## 1. 測試策略 (Testing Strategy)
基於我們所選擇的「快速原型開發」技術棧，本專案的測試策略旨在以最高效率驗證核心功能，確保應用程式的穩定性。

### 1.1 單元測試 (Unit Testing)

*  目的： 驗證後端程式碼中最小單位的邏輯正確性。

*  範圍：
    *  文件處理： 測試 src/utils/file_handler.py 中處理 .docx 檔案的函式，確保其能正確地讀取文本內容，並將修改後的內容存回。
    *  LLM 邏輯： 測試 src/utils/gemini_client.py 中 LangChain 記憶體模組的行為，確認多輪對話的上下文能夠被準確地捕捉和管理。

*  工具： 建議使用 Pytest 框架，其語法簡潔且易於上手。

### 1.2 LLM 功能測試 (LLM Functionality Testing)

*  目的： 評估 AI 模型的實際回覆品質與行為是否符合預期。

*  方法： 由於此類測試難以完全自動化，將以手動測試為主，並建立測試案例清單：
    *  讀者 AI： 準備不同風格的小說段落，輸入給 AI 後，人工評估其評價是否符合情境、有建設性。
    *  編劇 AI： 透過模擬多輪對話，驗證 AI 是否能記住先前的討論，並持續提供具備上下文連貫性的情節建議。

### 1.3 使用者介面測試 (UI Testing)

*  目的： 確保應用程式的使用者介面 (UI) 和互動流程順暢無誤。

*  方法： 採用手動探索性測試。
    *  檔案處理流程： 測試從上傳檔案、在編輯器中修改、到最終儲存的完整流程。
    *  聊天介面： 測試在網路連線不穩定時，AI 回應區是否能正確顯示載入狀態或錯誤訊息。
    *  複製功能： 驗證使用者能夠順利從 AI 回應區複製文字，並貼到小說編輯區。


## 2. 部署方案 (Deployment Plan)
由於您沒有預算且無法使用信用卡，所有雲端部署方案皆不適用。因此，本專案的部署方式將完全基於本地端運行。

### 2.1 部署方式 (Deployment Method)

*  唯一選擇： 本地端運行 (Local-Only)

*  優勢：
    *  零費用： 不產生任何雲端運算或儲存費用。
    *  最高資料隱私： 所有使用者資料（小說內容、對話紀錄）皆完全保存在使用者本機電腦上。

*  限制：
    *  應用程式無法公開給網際網路上的其他人使用，僅能作為個人工具在單機上運行。

### 2.2 本地端運行流程 (Local Execution Process)

為了讓使用者能在本地端順利運行應用程式，您需要在專案中提供清晰的指引。此流程應包含在專案的 README.md 文件中。

#### 1. 環境安裝：

*  前提條件： 使用者的電腦上需安裝 Python 3.8 或更高版本。
*  專案依賴： 指導使用者開啟終端機，在專案根目錄執行以下命令來安裝所有必要的函式庫：
    pip install -r requirements.txt

#### 2. API 金鑰配置：

*  目的： 應用程式需要使用者的 Google Gemini API 金鑰來呼叫模型。

*  操作： 指導使用者在專案根目錄下建立一個 .env 檔案，並在其中設定金鑰。
    GEMINI_API_KEY="你的 API 金鑰"

#### 3. 執行應用程式：

*  命令： 指導使用者在終端機中，執行以下命令來啟動 Streamlit 應用程式：
    streamlit run app.py
*  結果： 應用程式將自動在瀏覽器中開啟，使用者即可開始使用。

此部署方案確保了您能在零成本的情況下，順利地將專案從開發階段轉換為可執行的應用程式。
